# SEO: OG Meta Tags and Static Pre-rendering

## Goal

Make every view share a meaningful title, description, and preview image when pasted into Slack, Discord, Twitter, or any Open Graph-aware platform.

---

## Why dynamic JS meta tags aren't enough

The first attempt used `router.beforeEach` to update `og:title`, `og:description`, and the rest via `document.querySelector` on every navigation. That works for users navigating inside the app — the tab title and in-document tags update correctly — but social media crawlers don't execute JavaScript. They fetch the raw HTML, read the `<head>` once, and move on. So every URL shared would show the same generic default.

The fix is to pre-render a static `index.html` for each route with the correct tags already baked in, served directly by the file server before any JS runs.

---

## Architecture

```
pnpm build
  └── vue-tsc --noEmit          # type-check
  └── vite build                # bundle the SPA into dist/
  └── node scripts/generate-screenshots.mjs   # Playwright screenshots → dist/previews/
  ;   node scripts/generate-route-html.mjs    # inject meta tags into per-route HTML files
```

The `;` between the two scripts means HTML generation always runs even if screenshots fail (e.g., CI environment without a display).

---

## Route discovery

Both scripts share a route discovery module at `scripts/routes.mjs`. It scans `src/views/{Group}/` with the same logic as the Vite-based router config (`src/config/router.ts`), but runs in plain Node.js with no Vite or TS dependencies. Each entry returns `{ path, name, group, viewName }`.

---

## Static HTML generation

`scripts/generate-route-html.mjs` reads `dist/index.html` and, for each route, replaces the meta tag `content` attributes with route-specific values using regex, then writes the result to `dist/{path}/index.html`.

```
dist/
  index.html                          ← default (app shell)
  experiments/
    MaterialsList/
      index.html                      ← og:title = "Materials List | Generative Art"
    EarthGazer/
      index.html
  games/
    MazeGame/
      index.html
  ...
```

Netlify (and any static file server) serves the per-route file when a crawler hits that URL directly. Client-side navigation still uses the main bundle — only the initial HTML differs.

**Image resolution order:**

1. Explicit `image` field in `viewsMeta.json`
2. Screenshot at `dist/previews/{group}/{viewName}.png` (generated by Playwright)
3. `/logobw.svg` fallback

---

## Playwright screenshot generation

`scripts/generate-screenshots.mjs`:

1. Starts a local HTTP server on port 4321 serving `dist/`
2. Launches headless Chromium via Playwright
3. For each route, opens a 1200 × 630 viewport, navigates, waits for `networkidle` + 3 s render time, then takes a screenshot
4. Saves to `dist/previews/{group}/{viewName}.png`
5. Routes are processed in batches of 5 to avoid overloading the browser

The 3-second wait covers Three.js scene initialisation (Rapier WASM init, texture loading, first render frame). Complex views with model loading may capture a partially-loaded state — for those, add an explicit `image` path in `viewsMeta.json` pointing to a manually-crafted preview.

**Graceful failure:** if Playwright browsers are not installed (e.g., a CI environment without `playwright install`), the script catches the error, prints a warning, and exits with code 0 so the HTML generation still runs with the logo fallback.

---

## Per-view metadata

`src/config/viewsMeta.json` is the single source of truth for descriptions and optional custom images. Both the TypeScript app (`src/config/viewsMeta.ts`) and the Node.js build scripts read this JSON directly, so there is no duplication and no Vite/TS dependency in the scripts.

To customise a view:

```json
{
  "Materials List": {
    "description": "Interactive showcase of all Three.js PBR material types with live property controls.",
    "image": "/previews/custom/materials-list.png"
  }
}
```

---

## Key lessons

- **Crawlers don't run JS.** Dynamic meta tag updates via `router.beforeEach` are invisible to link-unfurl bots. Static HTML per route is the only reliable solution without SSR.
- **The `;` separator matters.** Using `&&` between the screenshot and HTML scripts would abort the build if screenshots fail. Using `;` keeps them independent, so the HTML step always produces valid output.
- **Batch Playwright pages.** Opening all 44 routes simultaneously (`Promise.all` over all routes) overloads the browser process. Batches of 5 keep memory and CPU usage manageable.
- **Route discovery duplication.** The router uses `import.meta.glob` which is Vite-specific. The build scripts need the same logic in plain Node.js. The shared `scripts/routes.mjs` module avoids repeating the logic in every script while staying dependency-free.
